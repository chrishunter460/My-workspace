# Build Pipeline & Hooks

> BaseModel -> DomainModel -> GraphBuilder -> NodeExecutor build chain, plus HookRunner and ArtifactStore.

## Purpose & Design Decisions

### What Problem This Solves

Domain models in de_Funk are defined as markdown files with YAML frontmatter that describe Bronze sources, dimension/fact schemas, joins, filters, derived columns, and build phases. The build pipeline translates these declarative configs into executable table-building operations: reading Bronze Delta tables, applying transformations (filter, join, select, derive, dedup, unpivot, window), and writing the resulting dimension and fact tables to Silver storage.

The challenge is that domain configs express intent declaratively (e.g. "this dimension comes from SELECT DISTINCT on these columns with these derived expressions") while the actual execution requires imperative DataFrame operations that differ between Spark and DuckDB backends. The build pipeline bridges this gap by providing a config-driven executor (NodeExecutor) that maps operation names to Engine methods, plus extension points (hooks, custom_node_loading) for logic that cannot be expressed in YAML alone.

### Key Design Decisions

| Decision | Rationale | Alternative Considered |
|----------|-----------|----------------------|
| Config-driven pipeline via NodeExecutor op registry | YAML nodes declare `from`, `pipeline` steps, and `select`. NodeExecutor maps each step's `op` field to an Engine method, making the build pipeline data-driven and backend-agnostic. New ops can be registered without modifying core code. | Hardcoded transformation chains per model |
| DomainModel handles special node types (seed, union, distinct, window) via `custom_node_loading` | These node types require complex multi-step logic (inline data creation, multi-source UNION, GROUP BY + derive, indicator computation) that cannot be expressed as a linear pipeline of ops. Keeping this in DomainModel rather than NodeExecutor preserves separation of concerns. | Expanding NodeExecutor with increasingly complex built-in ops |
| HookRunner: YAML config first, @pipeline_hook decorator fallback | Most hooks are simple side effects (logging, validation) that can be declared in YAML as `{fn: "module.function"}`. Complex hooks that return values (like `custom_node_loading` returning a DataFrame) use the decorator registry as an escape hatch. | All hooks in Python code (loses the config-is-king principle) |
| DomainBuilderFactory creates builder classes dynamically with `type()` | Avoids writing a separate Python builder class for each domain model. The factory scans `domains/models/` and creates builder classes at runtime, each with the correct `model_name`, `depends_on`, and config loading logic. | Manual builder class per domain model |

### Config-Driven Aspects

| Behavior | Controlled By | Location |
|----------|--------------|----------|
| Table sources (Bronze paths) | `graph.nodes.{id}.from` in domain YAML | `domains/models/{domain}/model.md` |
| Pipeline steps (filter, join, select, derive, dedup) | `graph.nodes.{id}.pipeline` in translated config | Generated by `config_translator.translate_domain_config()` |
| Build phase ordering | Node order in `graph.nodes` (sequential execution) | `domains/models/{domain}/model.md` |
| Hook functions (pre_build, after_build, post_build) | `hooks:` section in model YAML | `domains/models/{domain}/model.md` |
| Model dependencies | `depends_on:` in model frontmatter | `domains/models/{domain}/model.md` |
| Custom model class override | `CUSTOM_MODEL_CLASSES` dict (currently empty -- all use DomainModel) | `src/de_funk/models/base/domain_builder.py` |
| Indicator parameters for window tables | `_schema` column options with `{indicator: "sma", period: 20}` | Domain model schema definitions |

## Architecture

### Where This Fits

```
[Domain Markdown Configs] --> [DomainConfigLoader + ConfigTranslator]
                                        |
                                  [Translated Config]
                                        |
[DomainBuilderFactory] --> [BaseModelBuilder.build()]
                                        |
                              [BaseModel / DomainModel]
                                        |
                                  [GraphBuilder]
                                   /          \
                     [NodeExecutor]        [custom_node_loading]
                          |                       |
                    [Engine ops]          [seed/union/distinct/window]
                          |                       |
                     [dims + facts DataFrames]
                                        |
                              [Engine.write() --> Silver Delta]
```

The build pipeline sits between domain configuration (markdown YAML) and Silver storage (Delta Lake tables). It is invoked by `build_models.py` scripts and scheduled jobs. Upstream, `DomainConfigLoader` reads markdown configs and `ConfigTranslator` normalizes them into the graph node format. Downstream, the built Silver tables are consumed by the API layer's `FieldResolver` and DuckDB Engine for interactive queries.

## How to Use

### Common Operations

**Building a single domain model:**

```python
from de_funk.app import DeFunk

defunk = DeFunk.from_config("configs")
session = defunk.build_session(backend="spark")

# Get the builder for a specific model
from de_funk.models.base.builder import BuilderRegistry
from de_funk.models.base.domain_builder import discover_domain_builders

discover_domain_builders(Path("."))
builder_class = BuilderRegistry.all()["securities.stocks"]
builder = builder_class(session)
result = builder.build()
# BuildResult: model_name='securities.stocks', success=True, dimensions=2, facts=3, duration=12.4s
```

**Using NodeExecutor to run a custom pipeline:**

```python
from de_funk.core.executor import NodeExecutor

executor = NodeExecutor(session)
# Register a custom op
executor.register_op("my_transform", lambda df, step: my_fn(df))

# Execute a node config
result = executor.execute_node("my_table", {
    "from": "bronze.alpha_vantage.prices",
    "pipeline": [
        {"op": "filter", "conditions": ["trade_date >= '2024-01-01'"]},
        {"op": "select", "columns": ["ticker", "trade_date", "close"]},
        {"op": "derive", "expressions": {"return_pct": "(close - LAG(close) OVER (PARTITION BY ticker ORDER BY trade_date)) / LAG(close) OVER (PARTITION BY ticker ORDER BY trade_date)"}},
    ]
}, built={})
```

**Declaring hooks in domain model YAML:**

```yaml
# In domains/models/corporate/entity/model.md frontmatter:
hooks:
  after_build:
    - fn: de_funk.hooks.corporate.cik_enrichment.enrich_cik
      params:
        source: sec_cik_mapping
```

**Using the @pipeline_hook decorator for complex hooks:**

```python
from de_funk.core.hooks import pipeline_hook

@pipeline_hook("custom_node_loading", model="temporal")
def generate_calendar(engine, config, node_id, node_config, **kwargs):
    """Generate the temporal calendar dimension."""
    if node_id != "dim_calendar":
        return None
    # Return a DataFrame -- can't do this in YAML
    return engine.create_df(calendar_rows, schema)
```

### Integration Examples

**Build pipeline invoked by the scheduler:**

The `PipelineScheduler.job_daily_silver_rebuild()` function calls `build_model()` for each domain in dependency order. Internally this creates a `BuildSession`, discovers domain builders via `DomainBuilderFactory`, and calls `builder.build()` which triggers the full BaseModel -> GraphBuilder -> NodeExecutor chain.

**ArtifactStore for ML model lifecycle:**

```python
from de_funk.core.artifacts import ArtifactStore, ModelArtifact

store = ArtifactStore("storage/models")

# Save a trained model
artifact = ModelArtifact(
    model_name="stock_forecast",
    version="2026-03-01",
    metrics={"rmse": 0.042, "r2": 0.87},
)
store.save(artifact, trained_model_object)

# Load the latest version for prediction
artifact, model = store.latest("stock_forecast")
predictions = model.predict(features)
```

## Triage & Debugging

### Symptom Table

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| `BuildResult.error = "[CONFIG] ..."` | Domain markdown config has missing or malformed YAML frontmatter | Check the model's `model.md` for YAML syntax errors, missing `graph.nodes`, or invalid `from:` references |
| `BuildResult.error = "[STORAGE] ..."` | Bronze table path does not exist or Silver output path is not writable | Verify Bronze tables exist with `ls storage/bronze/`, ensure Silver directory is writable |
| `Window node 'xxx': source 'yyy' not yet built` | Window table references a source node that is in a later build phase | Move the source node to an earlier phase in the `graph.nodes` ordering |
| `Distinct table 'xxx' missing from or group_by` | The `__distinct__` node config is missing `_distinct_from` or `_distinct_group_by` | Check the domain config's `sources:` section — the config translator generates these from `transform: distinct` |
| `Hook after_build/fn_path failed: ModuleNotFoundError` | The hook function path in YAML points to a module that does not exist | Verify the `fn:` path is correct (e.g. `de_funk.hooks.corporate.cik_enrichment.enrich_cik`) |
| `Aggregate transform not yet supported for DuckDB` | DomainModel's `_build_distinct_node` with `_aggregate: true` only works with Spark backend | Use Spark backend for models with aggregate transforms, or refactor to use standard pipeline ops |
| Build produces 0 rows for a fact table | Source Bronze table is empty or filter conditions are too restrictive | Check Bronze data exists, inspect `graph.nodes` filters, look for `Skipping source` warnings in logs |

### Debug Checklist

- [ ] Run the build with verbose logging: `python -m scripts.build.build_models --model <name> --verbose`
- [ ] Check `BuildResult.error` string for the error category prefix: `[CONFIG]`, `[STORAGE]`, or `[MODEL]`
- [ ] Inspect the translated config: add `logger.debug(f"Translated config: {model_config}")` in `get_model_config()`
- [ ] For each node, check the logs for `Executing node: <node_id>` messages and verify the source loaded
- [ ] For custom_node_loading issues, check the hook resolution order: YAML hooks -> @pipeline_hook decorators -> None
- [ ] Verify Bronze source tables exist: `ls storage/bronze/<provider>/<table>/`
- [ ] For Spark builds, check Spark UI at `http://localhost:4040` for failed stages
- [ ] After build, verify Silver output: `ls storage/silver/<domain>/dims/` and `storage/silver/<domain>/facts/`

### Common Pitfalls

1. **Node ordering matters**: GraphBuilder executes nodes sequentially. A node that references another node (e.g. `__distinct__` referencing a fact table) must appear after its source in `graph.nodes`. The config translator handles this for standard domain configs, but custom configs may get the order wrong.

2. **Custom node loading returns None vs raises**: Returning `None` from `custom_node_loading` tells GraphBuilder to handle the node normally. Raising an exception aborts the build. If your custom logic cannot handle a node, return `None` rather than raising.

3. **Hook context parameters**: Hook functions receive `config=model_cfg, engine=engine, model=model_instance` plus any extra kwargs. The `config` parameter is the translated model config dict, not the raw markdown. If you need raw config, load it separately.

4. **Backend-specific code in DomainModel**: Several methods in DomainModel (e.g. `_build_union_node`, `_build_distinct_node`) have Spark-specific branches using `unionByName` and DuckDB fallbacks using pandas. The DuckDB path is less mature. Prefer Spark for production builds.

5. **BuilderRegistry is a class-level singleton**: `BuilderRegistry._builders` persists across calls. In tests, call `BuilderRegistry.clear()` to reset state between test cases.
